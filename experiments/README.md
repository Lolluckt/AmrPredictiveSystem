# Експериментальне обґрунтування

Ця тека містить ізольований **discrete-event симулятор**, який кількісно
підтверджує ефективність запропонованої інформаційної системи моніторингу
і предиктивного обслуговування для парку AMR-роботів цехової виробничої
лінії.

## Чому окремий симулятор, а не Webots

Webots-симуляція повільна (real-time, з фізикою), запускається довго і
вимагає GUI. Для статистично значущого порівняння потрібно ~ ≥ 30 повторень
кожної стратегії з різними seed'ами — це ~3 години GUI-часу.

Discrete-event симулятор тут моделює ту саму систему (графи, місії,
батарею, доки, поломки), але:

- крок симуляції = 1 секунда модельного часу, ≈ 50 000× швидше за real-time;
- detерміністичний за seed-ом → повторювані експерименти;
- однакові вхідні параметри для обох стратегій → коректне A/B-порівняння;
- KPI пишуться у CSV → можна повторно обробляти.

Координати вузлів, мапи місій і пороги анімалій — **синхронізовані** з
живим контролером і backend-кодом (`amr_controller.py`, `seed.py`,
`anomaly.py`). Якщо бекенд змінюється — симулятор оновлюється, бо
підвантажує ті ж pure-Python мапи.

## Що порівнюємо

| Сторона        | Стратегія розподілу зарядки | Політика ТО                                  |
|----------------|------------------------------|----------------------------------------------|
| **Baseline**   | Greedy nearest (старе)       | Корективна: ремонт після поломки             |
| **Proposed**   | Centralised dock allocator   | Предиктивна: правила-аномалії → заявка завчасно |

## KPI (метрики)

Для кожного прогону логуємо подієво, далі агрегуємо:

| Метрика                          | Формула / визначення                                                     |
|----------------------------------|--------------------------------------------------------------------------|
| `availability_pct`               | % часу робот не у `failure` і не чекає вільного дока                     |
| `missions_completed`             | Скільки повних місій виконав робот за зміну                              |
| `dock_queue_time_s_avg`          | Серед. секунд від `low_soc` до фактичного початку зарядки                |
| `dock_collisions`                | Скільки разів два роботи опинились у радіусі 1 м від одного дока        |
| `empty_travel_to_dock_m_avg`     | Сер. метрів проїхано до дока                                             |
| `mtbf_h`                         | Mean Time Between Failures (год)                                         |
| `mttr_h`                         | Mean Time To Repair (год); коротший якщо є завчасний тікет               |
| `unplanned_downtime_pct`         | % часу робот у незапланованому простої (поломка без передбачення)        |
| `predictive_lead_time_min_avg`   | Сер. хвилин між першою аномалією і фактичною поломкою                    |
| `false_positive_rate`            | Частка аномалій, після яких поломка так і не сталась за 1 год             |

## Як запустити

```bash
cd experiments
pip install pandas matplotlib python-docx --break-system-packages
python run_experiment.py --runs 30 --shift-hours 8 --seed 42 --out results/
python report.py results/ --out звіт_експеримент.docx
```

`run_experiment.py` ганяє 30 прогонів × 2 стратегії = 60 шифт-симуляцій,
дампить `results/runs.csv` (один рядок = один прогон) і генерує графіки.

`report.py` бере CSV і складає Word-документ з таблицями, діаграмами і
двома статистичними тестами (Welch t-test + Cohen's d) для кожної метрики
— готово для розділу 4 диплома.

## Що показуєш на захисті

1. **Слайд з таблицею** — Baseline vs Proposed, всі метрики ± довірчий
   інтервал 95 %.
2. **Бар-діаграми** — найбільш виграшні метрики (доступність, продуктивність,
   простій).
3. **Time-series** — приклад одного прогону: видно, як predictive детектує
   поломку за 12-25 хв до неї і запобігає простою.
4. **Висновок** — кількісно: "запропоноване рішення підвищує доступність
   флоту на X %, скорочує MTTR на Y %, зменшує черги на доках з N до 0".

## Структура коду

- `sim.py` — discrete-event симулятор (~600 рядків, без зовнішніх залежностей).
- `metrics.py` — обчислення KPI з event-log.
- `strategies.py` — реалізація двох стратегій (dock allocator + maintenance policy).
- `run_experiment.py` — orchestrator: запускає матрицю seed × strategy.
- `report.py` — генератор Word-звіту з таблицями і chart'ами.
- `results/` — куди пишуться CSV + PNG (створюється на льоту).
